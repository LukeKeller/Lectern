    Implementation Plan

    Work breakdown for the open-source Readwise Reader clone: twelve deliverables (D0–D11), each with concrete steps and a "done when" gate. Companion to the architecture plan. Use the per-section feedback buttons to comment before we name the project and scaffold the repo.

    How to read this

    Each deliverable is an independently verifiable chunk of work with a clear acceptance gate. Deliverables map onto the architecture roadmap phases but are sliced for execution: D0–D8 get us to a deployed MVP; D9–D11 are post-MVP parity, enhancements, and the deferred EPUB/native-Android work.

    • Depends on — what must be done first.
    • Track — Setup, Backend, Frontend, Deploy, or Future. Backend and Frontend tracks run in parallel after the contract (D2) is frozen.
    • Steps — the checklist to execute (checkable in your browser).
    • Done when — the observable acceptance criterion. No step is "done" on a scaffold that compiles; it is done when the behavior is verifiable.
    Execution model
    Once the repo exists, deliverables are dispatched to parallel subagents per track (backend vs frontend), coordinating on the frozen D2 contract. The frontend track (D5–D6) develops against a mock server generated from the D2 OpenAPI spec, so it does not block on the real BFF (D3–D4).

    Decisions to lock before D0
    Four things gate the scaffold: (1) repo host — GitLab (your default) works for a custom YunoHost app installed via git URL; the official catalog is GitHub-based but optional. (2) License — AGPL-3.0-only or MIT. (3) Single-user vs multi-user from day one (single is far simpler). (4) Dev environment for the two backends — Docker Compose locally is fine even though production is native YunoHost. These are surfaced again at D0/D1.

    D0 · Repo & toolchain

    Phase 0 Track: Setup · Depends on: —

    Done when
    pnpm install && pnpm build && pnpm test is green in CI on the empty scaffold, and the first ADR is committed.

    D1 · Backend provisioning & API spike

    Phase 0 Track: Setup · Depends on: D0

    Done when
    A short spike report exists, progress + highlight writes round-trip against both APIs, and the read-later backend decision is recorded as an ADR.

    D2 · Shared domain model & contracts

    Phase 1 Track: Setup · Depends on: D1 · freezing this unblocks both tracks

    Done when
    The shared package builds and exports types, the OpenAPI spec validates, the query AST round-trips in tests, and a mock server serves the contract.

    D3 · BFF data layer & backend adapters

    Phase 1 Track: Backend · Depends on: D2

    Done when
    Given seeded feeds + a saved article, the unification service returns a correct merged Card list, and location/progress/tag writes land in the right store — verified by integration tests against live dev backends.

    D4 · BFF API, auth & jobs

    Phase 1 Track: Backend · Depends on: D3

    Done when
    The full unified API passes route tests (including auth failures), new feed items sync into the Card list within the poll interval, and health endpoints report ready.

    D5 · PWA shell & data layer

    Phase 1 Track: Frontend · Depends on: D2 (runs in parallel with D3/D4 via the mock server)

    Done when
    The app boots offline with cached data, pulls deltas when online, and replays queued offline mutations on reconnect — verified by toggling the network.

    D6 · Reading experience

    Phase 1 Track: Frontend · Depends on: D5

    Done when
    A saved article and an RSS item both read end-to-end with progress restored across a reload, triage moves persist, and tags/search/views work offline.

    D7 · Save flows

    Phase 1 Track: Full-stack · Depends on: D4, D6

    Done when
    Saving via bookmarklet, Android share target, manual URL, and RSS→read-later each produce a readable Card, and OPML import creates feeds.

    D8 · YunoHost packaging & deploy

    Phase 1 Track: Deploy · Depends on: D4, D6, D7

    Done when
    yunohost app install <git_url> yields a working SSO-gated app + token API on the VPS, backup/restore/upgrade pass, and it shows in the webadmin Services page. This is the MVP milestone.

    D9 · Parity features

    Phase 2 Track: Full-stack · Depends on: D8

    Done when
    Highlights round-trip on both sources, arbitrary saved-view queries return correct result sets, and a forwarded newsletter appears in the Feed.

    D10 · Enhancements

    Phase 3 Track: Full-stack · Depends on: D9

    Done when
    Each feature is usable behind a settings toggle without regressing the core reading/triage flows.

    D11 · Deferred follow-ups

    Phase 4 Track: Future · Depends on: D10

    Done when
    A book opens from OPDS with reading position syncing to/from KOReader, and an Android build installs from an APK/AAB.

    2. Confirm the four D0 decisions: repo host, license, single- vs multi-user, dev-backend setup.
    3. Scaffold the repo (D0) and stand up the dev backends (D1).
