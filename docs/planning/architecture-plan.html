    Open-Source Readwise Reader Clone

    A self-hosted Readwise Reader alternative for RSS + read-later, assembled from MiniFlux (feeds) and Readeck (saved articles) behind a custom aggregator, delivered as an installable PWA and website, packaged as a native YunoHost app. EPUB reading is explicitly deferred.

    Executive summary

    Readwise Reader's value is not any single backend feature — it is the unified experience: one inbox spanning RSS and saved articles, one triage workflow, cross-device reading progress, highlights, tags, saved smart-views, and offline full-text search. No single open-source tool reproduces that. The pragmatic build is therefore a thin unification layer over two best-in-class self-hosted backends:

    • MiniFlux — the RSS half. Single Go binary + Postgres, tiny footprint, excellent token-auth REST API, full-content scraping, webhooks, OPML. Already in the YunoHost catalog.
    • Readeck — the read-later half. Single Go binary + Postgres, and uniquely among the candidates it exposes writable reading progress (read_progress + read_anchor) and highlights/annotations over a clean OpenAPI Bearer-token API. Already in the YunoHost catalog.

    We build exactly one new component: a TypeScript aggregator (backend-for-frontend) + PWA that owns the cross-source state the backends cannot (unified location/triage, unified tags, saved views, RSS-item progress/highlights), serves an offline-first reading UI, and is packaged as a single native YunoHost app. Docker is avoided throughout — everything is native, catalog-clean, and SSO-integrated.

    Headline recommendation
    MiniFlux (RSS) + Readeck (read-later) as backends, both installed from the YunoHost catalog. One custom SvelteKit PWA + Node/Fastify BFF + small Postgres app for unification, packaged natively for YunoHost with SSO. Readeck is chosen over Wallabag specifically because it is the only option that syncs reading progress and highlights through its API — collapsing the hardest part of the clone into the backend.

    Goal & scope decisions

    In scope: RSS feeds and saved-articles ("read later") with a Readwise-style triage, reading, annotation, and search experience, on web + Android (installable PWA acceptable in place of a native app), deployed on a YunoHost VPS.

    Reader API surface worth mirroring (target shape for our unified API)

    Base https://readwise.io/api/v3/, header Authorization: Token <t>. Endpoints: POST /save, GET /list (cursor-paginated, filters by location/category/tag/updatedAfter, optional withHtmlContent), PATCH /update/<id>, DELETE /delete/<id>, GET /tags. Document fields include id, url, source_url, title, author, category, location, tags, word_count, reading_time, reading_progress, first_opened_at, last_opened_at, summary, notes, html_content. Categories: article|email|rss|pdf|highlight|note|tweet|video|epub. Locations: new|later|shortlist|archive|feed (shortlist is filter-only, not writable). Our BFF should expose this same shape so any Reader-compatible tooling can talk to it.

    Why Readeck wins
    Reading-progress sync and highlights are the two hardest features to bolt on externally. Readeck ships both behind a clean token API, so for saved articles they become a backend concern, not ours. It is also a single Go binary + Postgres — the same operational shape as MiniFlux. Swap path: if extraction quality on your real sources (paywalls, Substack, JS-heavy SPAs) proves weak, Wallabag's FiveFilters site-config is stronger; the BFF abstracts the backend so swapping it later changes one adapter, at the cost of building progress-sync for saved articles yourself.
    Built-in synergy
    MiniFlux has a native "save entry to read-later" integration that pushes directly to Readeck (and Wallabag). So "save this RSS item to read later" can be a one-call backend operation. We prefer routing it through the BFF instead, so our unified tags/location ride along — but MiniFlux's integration is a working fallback.

    Components

    • PWA client — offline-first reading UI; local IndexedDB cache + client-side search; talks only to the BFF.
    • Aggregator BFF — normalizes both backends into the unified model, owns cross-source state, runs sync/poll/email jobs, exposes one Reader-style API + a sync delta API.
    • Glue Postgres — the only new datastore: unified location, unified tags + links, saved views, RSS-item progress + highlights, ingestion log, per-user backend tokens, sync cursors.
    • MiniFlux / Readeck — unmodified catalog installs; the BFF holds each one's API token and calls them server-side.

    The one swappable decision
    The client framework is the most replaceable choice — SvelteKit and Next.js/React both deliver an offline-first PWA. Everything else (two Go backends, a TS BFF, Postgres, native YunoHost packaging) is load-bearing and chosen for operational simplicity on a single VPS.

    The unified "card" the client renders, regardless of source:

  "source_id": "abc123",        // id within the backend
  "category": "article",        // article | rss | email | pdf
  "location": "later",          // inbox | later | shortlist | archive | feed
  "title": "…",
  "author": "…",
  "site_name": "…",
  "url": "https://…",           // original source
  "word_count": 1840,
  "reading_time": "8 min",
  "reading_progress": 0.42,     // 0..1  (Readeck native / BFF for RSS)
  "read_anchor": "#p-37",       // Readeck-rendered articles only
  "tags": ["ai", "longread"],   // unified (BFF)
  "highlight_count": 3,
  "saved_at": "2026-06-03T10:00:00Z",
  "updated_at": "2026-06-03T12:30:00Z"
}

    Sync & conflict rules

    • Pull (server→client): per-source delta cursors. MiniFlux GET /v1/entries?changed_after=… (+ new_entries webhook for push); Readeck list polling or its sync endpoint. The BFF folds both into one updatedAfter stream for the client.
    • Push (client→server): client queues mutations (location, progress, tags, highlights) while offline; BFF applies them to the owning backend or glue DB on reconnect.
    • Conflicts: last-write-wins per field, keyed by updated_at, scoped to the owning store from the source-of-truth table. No field has two writers, so conflicts are rare by construction.
    • Offline: client precaches article HTML + metadata for Inbox/Later/Shortlist; full-text search runs locally; all reading and triage work offline and replay on reconnect.

    Auth & YunoHost deployment

    All backends already ship official packaging-v2 YunoHost packages, so we install them as-is and package only the aggregator. Auth uses YunoHost's SSO for the web UI and an app-issued bearer token for the PWA/Android sync API — exactly the pattern MiniFlux/Readeck themselves use for their API paths.

    • Web UI behind the main permission (SSO); the BFF trusts the spoof-protected Ynh-User header for identity — no separate login screen.
    • Sync API behind a separate api permission with auth_header = false and the BFF's own per-user bearer token, so SSO redirects never break mobile clients.
    • Backend calls are server-side from the BFF using each backend's own API token (stored as masked app settings), never via SSO headers.
