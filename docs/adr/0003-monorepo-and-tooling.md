# 3. Monorepo and tooling

Date: 2026-06-03
Status: Accepted

## Context

Web (SvelteKit) and BFF (Fastify) share a domain model and an API contract. We want shared
types, one install, and consistent lint/format/test across a single deploy artifact.

## Decision

- **pnpm workspace monorepo**: `apps/web`, `apps/bff`, `packages/shared`, `packages/api-client`,
  plus `ynh/` and `infra/`.
- **TypeScript everywhere**, strict, `moduleResolution: bundler`. Workspace packages export
  their TS source directly (`exports → ./src/index.ts`); bundlers (Vite/tsup/Vitest) consume
  source, so no pre-build step is needed for dev/test. The BFF bundles workspace source via
  tsup (`noExternal: @lectern/*`).
- **Tooling**: ESLint (flat config) + Prettier; Vitest for unit tests. The web app keeps its
  own Svelte-aware ESLint/Prettier config (generated by `sv`); the root config covers the
  TypeScript packages.
- **Single-user** for v1 (one set of backend tokens; no per-user provisioning).
- Repo on **GitHub**; license **AGPL-3.0-only**.

## Consequences

- `pnpm -r {typecheck,test,build,lint}` fan out across packages in dependency order.
- Adding the generated API client (D2) drops into `packages/api-client` without restructuring.
- Two ESLint configs (root + web) is a deliberate trade for proper Svelte linting.
